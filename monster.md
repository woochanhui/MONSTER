# 🚀 GitHub Flow (브랜치 전략) 협업 가이드

## 📌 브랜치 운영 원칙
* **main**: 언제든 배포 가능한 완벽한 최종 버전 (직접 커밋 금지)
* **dev**: 팀원들의 작업물을 모아 테스트하는 중간 조립 구역 (직접 커밋 금지)
* **feature/기능명**: 각 팀원의 개인 기능 개발 공간 (작업 완료 후 dev로 PR)

---

## 🏃‍♂️ 개발자 하루 루틴 (명령어 흐름)

### 1단계: 작업 전 최신 코드 가져오기
새로운 기능을 만들기 전, 다른 팀원이 원격 저장소에 올린 최신 코드를 내 컴퓨터로 가져와 일치시킵니다.
$ git checkout dev
$ git pull origin dev

### 2단계: 오늘 작업할 기능 브랜치 생성 및 이동
최신화된 dev 브랜치를 바탕으로 나만의 독립된 작업 공간을 만듭니다. 브랜치 이름은 작업할 기능에 맞춰 자유롭게 지정합니다.
$ git checkout -b feature/기능명

### 3단계: 소스 코드 수정 및 로컬 커밋
내 기능 브랜치 안에서 코드를 작성하거나 파일을 수정하고, 의미 있는 작업 단위로 커밋을 쌓아나갑니다.
$ git add .
$ git commit -m "feat: 어떤 기능을 구현했는지 명시"

### 4단계: 원격 저장소(GitHub)에 내 브랜치 업로드
내 컴퓨터에만 존재하는 기능 브랜치를 다른 팀원들이 볼 수 있도록 깃허브 웹사이트로 전송(Push)합니다.
$ git push origin feature/기능명

---

## 🤝 깃허브 웹사이트에서의 작업 (코드 리뷰 & 머지)

### 1. Pull Request (PR) 생성
* 깃허브 저장소 페이지 상단의 [Compare & pull request] 버튼을 클릭합니다.
* 브랜치 병합 방향 확인 (필수): 반드시 base: dev ⬅️ compare: feature/기능명 상태여야 합니다. (main으로 바로 보내지 않도록 주의합니다.)
* 오늘 작업한 내용 요약과 리뷰어(팀원)를 지정한 뒤 [Create pull request]를 클릭합니다.

### 2. 코드 리뷰 및 승인
* 알림을 받은 팀원들이 PR 페이지의 Files changed 탭에서 코드를 확인하고 피드백을 남깁니다.
* 코드에 문제가 없다면 팀원이 우측 상단의 [Approve]를 눌러 승인합니다.

### 3. 최종 병합 (Merge)
* 최소 1명 이상의 승인을 받아 조건이 충족되면 PR 페이지 하단의 [Merge pull request]를 눌러 원격의 dev 브랜치에 최종 반영합니다.
* 머지가 완료되면 깃허브 웹 페이지에 뜨는 [Delete branch]를 눌러 역할을 다한 원격 브랜치를 지워줍니다.

---

## 🧹 작업 종료 후 로컬 저장소 청소 및 동기화

원격 dev 브랜치에 내 코드가 완전히 병합되었으므로, 내 컴퓨터(로컬)의 상태도 찌꺼기 없이 깔끔하게 정리해 줍니다.
$ git checkout dev
$ git pull origin dev
$git branch -D feature/기능명$ git fetch --prune

*이 단계까지 끝나면 터미널은 다시 아무것도 붙지 않은 깨끗한 (dev) 상태가 됩니다. 다음 기능을 개발할 때 다시 1단계부터 이 루틴을 반복하시면 됩니다.*
